## 2025-05-18 - Fix unauthenticated SSRF in NVIDIA proxy
**Vulnerability:** The Vercel serverless function `handleNvidiaProxy` exposed an endpoint that consumed the backend `NVIDIA_API_KEY` but lacked any authentication checks. This allowed unauthenticated, cross-origin abuse of the endpoint.
**Learning:** Serverless proxy routes that forward to billing-sensitive third-party APIs must always have identical authentication checks as internal RPC calls.
**Prevention:** Always wrap Vercel node function handlers in an `await requireAuthenticatedUser(req)` block before proceeding to process the request payload or secrets.

## 2026-06-19 - Prevent unauthenticated SSRF in AI proxy routes and Shell Command Injection
**Vulnerability:** Unauthenticated users could exhaust resources or potentially trigger SSRF attacks because AI endpoints (`handleAIGenerate`, `handleAIExplain`, `handleAILessonBlocks` in `src/server/api/aiHandlers.ts`) were missing an explicit authentication check on the user object after resolving the platform context. Additionally, `package.json` had a shell command injection vulnerability where environment variables within `fn:secrets` were not properly escaped (`$GEMINI_API_KEY` instead of `\"$GEMINI_API_KEY\"`).
**Learning:** Returning `undefined` for `userId` from context resolvers does not automatically block requests. Route handlers must explicitly evaluate the context (e.g. `if (credentials.credentialMode === 'platform' && !credentials.userId)`) and return `401 Unauthorized`. For package scripts, raw environment variables can lead to command injection if a shell expands them unsafely.
**Prevention:** Always follow the pattern of explicitly verifying the returned credentials object in downstream route handlers, especially when switching between authentication contexts (e.g. `platform` vs `byok`). Always wrap shell environment variable injections in escaped double quotes in npm scripts.
